## 1. 架构总览

Chorify 是一个**前后端一体的多租户 SaaS**，由三个可独立部署的部分组成：

```
                          ┌─────────────────────────────────────────┐
        浏览器 / 手机端 ── │  apps/web  (Next.js 16, App Router)        │
                          │  ─ 前端 UI（工作流 + 服务 + 管理）          │
                          │  ─ 登录鉴权后端（短信 + JWT cookie）        │ ✅
                          │  ─ 业务 API（组织/工程/云盘/分发 CRUD）     │ 🔜
                          │  ─ AI 代理（把 AI 请求转发给 ai-service）   │
                          └───────┬───────────────────────┬───────────┘
                                  │ Prisma                │ HTTP (server-only)
                                  ▼                       ▼
                    ┌──────────────────────┐   ┌────────────────────────────┐
                    │ PostgreSQL 16        │   │ apps/ai-service (FastAPI)   │
                    │ + Prisma 6 (@chorify/db) │ │ ─ 素材生产 /api/material   │ ✅骨架
                    └──────────────────────┘   │ ─ 合成量产 /api/compose    │
                    ┌──────────────────────┐   │ ─ 翻译/切片/拆分… 🔜        │
                    │ Redis 7              │   │ ─ mock_mode：真/假可切换    │
                    │ 验证码/限流/缓存/队列 │   └───────────┬────────────────┘
                    └──────────────────────┘               │ 调用
                    ┌──────────────────────┐   ┌───────────▼────────────────┐
                    │ 阿里云 OSS（文件）    │   │ AI 供应商（可插拔）          │
                    │ 阿里云短信（登录码）  │   │ LLM / 文生视频 / TTS / 数字人 │
                    └──────────────────────┘   └─────────────────────────────┘
```

**职责切分原则**：
- **Next.js**：一切"轻"后端——登录鉴权、会话、业务 CRUD、对前端聚合数据；**不直接调 AI 供应商**，而是经 AI 代理转给 ai-service。
- **FastAPI（Python）**：一切"重"AI——生成/混剪/翻译/切片，便于用 Python AI 生态、独立扩缩容、长任务编排。
- **Postgres**：业务事实数据（强一致）。**Redis**：易失/高频数据（验证码、限流、缓存、任务队列）。
- **OSS**：所有二进制（视频/图片/音频/成片）。Postgres 只存 URL 与元数据。

## 4. 前端（Next.js 16）

### 4.1 路由分组
- `(auth)`：公开页（登录）。
- `(app)`：受保护应用区，统一 `layout.tsx`（侧边栏来自 `nav.ts`）。
- 占位模块统一用 `components/layout/ModulePlaceholder.tsx`，按路线图替换为真实页面。

### 4.2 路由保护（✅ `src/proxy.ts`，Next 16 "proxy" 约定，原 middleware）
- Edge 运行时**只做无状态 JWT 校验**（不碰 DB/Redis）。
- 无会话且非公开路径 → 重定向 `/login?next=...`；已登录访问 `/login` → 回首页。
- `matcher` 排除 `api/auth`、`_next` 静态资源。

### 4.3 数据获取
- 列表/详情优先用 **Server Component 直连 Prisma**（如首页 `RecentProjects` 已这样做），减少 API 往返。
- 需要客户端交互/写操作的走 `app/api/*` 的 Route Handler。
- **AI 长任务**：前端提交 → 拿 `job_id` → 轮询（或后续 SSE/WebSocket）状态。

### 4.4 品牌单点
- 产品名只在 `src/lib/brand.ts`（`NEXT_PUBLIC_APP_NAME`）。换名只改一处。

## 5. 鉴权与会话（✅ 已实现）

### 5.1 登录流程（无密码 · 手机号 + 验证码）
```
POST /api/auth/send-code  {phone}
   → 生成 6 位码，存 Redis（含 TTL + 频率限制），经 SMS provider 发送（dev=mock 打印控制台）
POST /api/auth/verify     {phone, code}
   → 校验 Redis 中的码 → upsert User → 加入演示组织（Membership）
   → signSession(JWT) 写入 httpOnly cookie（SESSION_COOKIE='chorify_session'）
GET  /api/auth/me         → 返回当前用户/组织
POST /api/auth/logout     → 清除 cookie
```
- **JWT**：`lib/auth/jwt.ts`（jose 签名/校验，载荷 `{userId, phone, orgId}`，TTL 由 `SESSION_TTL_DAYS`）。
- **Cookie**：httpOnly + sameSite=lax + 生产 secure（`lib/auth/session.ts`）。
- **当前用户**：`lib/auth/current-user.ts` 供 Server Component 读取。

### 5.2 Redis 在鉴权中的职责（`lib/redis.ts`，ioredis）
- 验证码存取与过期；同手机号发送频率限制；登录尝试限流。

### 5.3 安全要点
- 验证码：长度/有效期/尝试次数/同号频控，防爆破。
- JWT 密钥 `JWT_SECRET` ≥ 32 字符，生产从密管注入。
- 所有业务查询带 `orgId`，防越权读他组织数据（多租户隔离）。

## 6. 数据层（Prisma 6 + PostgreSQL）

### 6.1 已实现模型（`packages/db/prisma/schema.prisma`）
| 模型 | 用途 | 关键字段 |
|---|---|---|
| `User` | 用户 | phone(unique), nickname, lastLoginAt |
| `Organization` | 租户/组织 | name, plan(FREE/PRO/ENTERPRISE) |
| `Membership` | 用户↔组织 | role(OWNER/ADMIN/MEMBER), unique(userId,orgId) |
| `Project` | 工程/合成量产 | type(SMART_MIX/SUPER_MIX/ONE_CLICK), status, comboCount, width/height |
| `Folder` | 云盘文件夹 | 自引用 parentId（树）|
| `Asset` | 云盘文件 | type(VIDEO/IMAGE/AUDIO/PROJECT/FINISHED/MATERIAL), url, durationSec |
| `Distribution` | 投放分发 | platform, accountCount, videoCount, publishDone/Total, views, likes |

枚举：`Role` `Plan` `ProjectType` `ProjectStatus` `AssetType` `AssetStatus` `Platform`。
索引：列表查询均已建 `@@index([orgId, ...])`。映射：`@@map` 到复数蛇形表名。

### 6.2 约定
- 主键 `cuid()`；`createdAt`/`updatedAt` 标配；删除用状态（`AssetStatus.DELETED`）做软删，重要数据不物理删。
- 客户端单例从 `@chorify/db`（`packages/db/src/index.ts`）导出，`apps/web` 经 `lib/db.ts` 使用。

### 6.3 待扩展模型（🔜 支撑全量产品，建议新增）
> 以下为路线图中后续模块所需，先记录设计，落地时再迁移。

```prisma
// AI 异步任务（素材生产/混剪/翻译/切片统一）
model Job {
  id          String   @id @default(cuid())
  kind        JobKind  // MATERIAL | MIX | TRANSLATE | LIVE_CLIP | SPLIT
  status      JobStatus // QUEUED | RUNNING | SUCCEEDED | FAILED
  provider    String?   // seedance / tts / digital-human ...
  input       Json
  resultUrl   String?
  error       String?
  projectId   String?
  orgId       String
  createdById String
  creditsCost Int       @default(0)
  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt
  @@index([orgId, status])
}

// 矩阵宝：被托管的社媒账号（投放分发的账号来源）
model SocialAccount {
  id        String   @id @default(cuid())
  platform  Platform
  handle    String
  status    String   // active | expired | banned
  groupName String?
  orgId     String
  @@index([orgId, platform])
}

// Credits 账本（用量计费）
model CreditLedger {
  id        String   @id @default(cuid())
  orgId     String
  delta     Int      // +充值 / -消耗
  reason    String   // mix / dubbing / digital_human ...
  jobId     String?
  createdAt DateTime @default(now())
  @@index([orgId, createdAt])
}
```
> `Project`/`Asset`/`Distribution` 在落地真实任务时，与 `Job` 建立关联（如 `Asset.jobId`、`Distribution` 关联 `SocialAccount`）。

## 7. AI 服务（FastAPI）

### 7.1 现状（✅ 骨架，`apps/ai-service`）
- `main.py`：FastAPI 应用 + CORS + `GET /health`（含 `mock_mode`）。
- `core/config.py`：pydantic-settings；`mock_mode=true` 时所有 AI 接口返回占位结果。
- 路由：
  - `material.py`：`POST /api/material/generate`、`GET /api/material/jobs/{id}`（type: ai_studio/replica/element_swap/dubbing/digital_human）。
  - `compose.py`：`POST /api/compose/mix {project_id,count}` → N 个 combo。

### 7.2 统一 Job 模型（建议固化为服务规范）
所有重任务遵循同一形态：
```
POST  /api/<domain>/<action>   提交 → {job_id, status:"queued"}
GET   /api/<domain>/jobs/{id}  轮询 → {job_id, status, result_url?}
（后续）回调/SSE 推送进度
```
- 任务入 **Redis 队列**，由 worker 消费（CPU/GPU 密集可独立扩缩容）。
- 完成后产物落 **OSS**，写回 **Postgres**（`Job` + 对应 `Asset`/`Project`）。

### 7.3 供应商可插拔（成本与替换的关键）
- 定义 `Provider` 接口（如 `VideoGenProvider`/`TTSProvider`/`DigitalHumanProvider`），mock 与各真实供应商各一实现。
- `mock_mode` 或配置决定注入哪个实现；**前端契约不变**，real 化只改服务内部。
- 多供应商可路由（按成本/质量/可用性），实现"成本工程"。

### 7.4 待建路由（🔜）：`translate.py` · `live.py`（直播切片）· `split.py`（视频拆分）· `inspiration.py`（LLM 脚本/解构）。

## 8. Web ↔ AI 通信（AI 代理）

- 前端**不直接**访问 ai-service。由 Next.js 服务端（Route Handler）作 **AI 代理**：注入鉴权/组织上下文、做配额/Credits 校验、再转发 `AI_SERVICE_URL`（见 `.env`）。
- 好处：ai-service 不暴露公网、统一鉴权与计费扣减、便于限流与审计。

## 11. 非功能需求

- **性能**：列表查询走索引 + 分页；首页等读多场景用缓存（Redis）；AI 长任务异步不阻塞。
- **可扩展**：web 与 ai-service 各自水平扩展；AI worker 按队列长度扩缩；OSS 承载大文件流量。
- **安全**：多租户 orgId 隔离、httpOnly+secure cookie、密钥外部注入、验证码频控、生成内容审核（合规）。
- **可观测**：结构化日志、任务全链路 traceId、关键指标（任务时延/成功率/单条成本）打点。
- **可靠**：任务幂等 + 重试 + 死信；外部供应商超时/熔断/降级（自动切备用 provider）。

## 12. 环境与部署

- **本地**：`docker compose up -d`（PG+Redis）→ `pnpm install` → `pnpm db:generate/migrate/seed` → `pnpm dev`（web :3000）；ai-service 用 `uvicorn` 起 :8000。
- **环境变量**：以 `.env.example` 为准，分发到 `apps/web/.env`、`packages/db/.env`、`apps/ai-service/.env`。
- **生产（建议）**：web 与 ai-service 容器化；托管 Postgres + Redis；OSS/短信用阿里云生产凭证；CI 跑 `turbo build/typecheck`；密钥走密管。

---

## 13. 编码与协作约定

- TypeScript 严格模式；服务端逻辑集中在 `lib/`，组件按 brand/home/layout/login/ui 归类。
- 数据库改动一律走 Prisma migration（`pnpm db:migrate`），禁止手改库结构。
- 新增模块 = 一条路由页 + （需要时）一组 `/api/*` + （重 AI）一个 FastAPI 路由，遵循统一 Job 模型。
- mock 与 real 严格分离在 provider 实现内，业务/前端不感知。
